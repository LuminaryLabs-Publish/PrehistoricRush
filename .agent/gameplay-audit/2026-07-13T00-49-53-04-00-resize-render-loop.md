# Gameplay Audit: Resize and Render Loop

**Timestamp:** `2026-07-13T00-49-53-04-00`

## Summary

Simulation and streaming continue independently of viewport validity. A resize has no gameplay/session identity and can mutate camera and renderer state between committed simulation frames without producing a typed result or a correlated visible frame.

## Plan ledger

**Goal:** keep gameplay truth independent from resize delivery while making presentation changes explicit and observable.

- [x] Trace RAF, simulation, streaming, render and HUD ordering.
- [x] Trace resize mutation outside the RAF transaction.
- [x] Identify missing surface revision in render and diagnostics.
- [x] Define a presentation-only commit boundary.
- [ ] Implement later without moving gameplay truth into rendering.

## Current loop

```txt
RAF
  -> input
  -> engine.tick
  -> committed RunState / physics / simulation
  -> streaming update
  -> camera follow
  -> renderer.render
  -> HUD

window resize callback
  -> camera aspect mutation
  -> projection-matrix mutation
  -> renderer size mutation
```

## Finding

The resize callback is not sequenced with the simulation/render transaction. It may run between frames or between other browser callbacks, but no viewport generation tells the next frame which camera and drawing buffer it consumed. Gameplay remains authoritative, yet visible presentation provenance is incomplete.

## Required behavior

```txt
resize observation
  -> prepare viewport candidate
  -> commit presentation revision
  -> next RAF consumes committed revision
  -> renderer applies it once
  -> visible-frame acknowledgement records simulation tick + viewport revision
```

No gameplay state should be rewound or mutated by a viewport failure. A rejected or zero-size candidate should defer rendering or retain the predecessor surface under an explicit result.
